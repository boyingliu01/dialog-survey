## 1. 问题陈述

当前管理后台 UI 将「访谈模板」、「访谈计划」、「访谈报告」作为三个并列的顶部导航入口，但实际数据模型存在明确的层级引用关系：

```
模板 (Template)
  └── 计划 (InterviewPlan) — 一个模板下可挂多个计划
        └── 受访者 (Interview) — 一个计划包含多个受访者
              └── 个人报告 (AnalysisReport) — 每个受访者一份报告
        └── 计划总结 (BatchAnalysisReport) — 每个计划一份汇总
```

**用户痛点**:
- 无法从模板直接看到关联了哪些计划
- 无法从计划直接看到哪些受访者已完成、哪些还在进行中
- 报告列表是平的，无法按计划/模板维度筛选
- 缺少跨计划的统计分析视图

---

## 2. 设计目标

1. **层级可见**: 导航结构反映真实数据关系
2. **下钻能力**: 从模板 → 计划 → 受访者 → 报告，逐级下钻
3. **状态一目了然**: 计划完成情况、人员进度在详情页清晰展示
4. **公共属性统一**: 所有实体包含 createdBy, createdAt, updatedBy, updatedAt

---

## 3. 导航结构

### 3.1 顶部导航 (保持不变，但行为变化)

```
首页 │ 模板管理 │ 访谈计划 │ 访谈报告 │ 统计分析(新) │ 系统设置
```

### 3.2 导航行为变更

| 入口 | 现状 | 变更后 |
|------|------|--------|
| 模板管理 | 列表页 + 内联编辑 | 列表页不变，点击模板名进入**模板详情页**(带 Tab) |
| 访谈计划 | 列表页，只读 | 列表页增加模板筛选 + 点击进入**计划详情页**(带 Tab) |
| 访谈报告 | 列表页，按时间排序 | 增加计划/模板/状态筛选，支持分组查看 |
| 统计分析 | 不存在 | **新增**: 跨计划汇总 Dashboard |

## 5. 实体属性定义

所有实体统一包含以下审计字段:

| 字段 | 类型 | 说明 |
|------|------|------|
| `createdBy` | String | 创建人标识 (当前为 `admin`，未来可扩展) |
| `createdAt` | DateTime | 创建时间 (自动) |
| `updatedBy` | String | 最后修改人标识 |
| `updatedAt` | DateTime | 最后更新时间 (自动) |

## 6. 数据流

```
用户操作:
  创建模板 → 维护问题和 Prompt
       ↓
  从模板创建计划 → 添加受访者列表
       ↓
   DingTalk 发起访谈 → 受访者通过钉钉对话完成
       ↓
  生成个人报告 (AnalysisReport)
       ↓
  计划全部完成 → 生成计划总结 (BatchAnalysisReport)
       ↓
  统计分析页聚合展示
```

## 7. 技术约束

1. **现有 HTMX + Nunjucks 架构保持不变**
2. **Nunjucks 内置过滤器**: `truncate(80, true)` 已用于内容截断
3. **Prisma 关系**: `Template 1:N InterviewPlan`, `InterviewPlan 1:N Interview`, `Interview 1:1 AnalysisReport`
4. **无需新增数据库表**: `AnalysisReport` 已存在，`BatchAnalysisReport` 已存在
5. **`ADMIN_API_KEY`** 环境变量必须配置 (已有)

## 9. 决策记录

| 决策 | 理由 |
|------|------|
| 保持顶部导航，不改为左侧树 | 数据仅 2 层深，树形导航过重 |
| 详情页使用 Tab 而非多页面 | 减少页面跳转，上下文保留 |
| 报告列表保持独立入口 | 快速查看所有报告，无需逐层下钻 |
| 统计分析页单独入口 | 跨维度聚合，不属于任何单一实体 |
| 手动生成计划总结 | 用户确认后再执行，避免不必要的 LLM 调用 |
