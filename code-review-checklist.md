# 代码审查清单 (Code Review Checklist)

> 本清单用于 PR 提交前的自我检查，确保代码质量符合标准。

---

## 🎯 Clean Code 检查

### 命名规范

- [ ] 变量名是否清晰表达意图？
- [ ] 函数名是否描述其行为？
- [ ] 类名是否使用名词？
- [ ] 避免使用缩写和单字母变量（循环变量除外）
- [ ] 布尔变量使用 `is_`, `has_`, `can_` 前缀

### 函数规范

- [ ] 函数是否短小（建议 < 20 行）？
- [ ] 函数是否只做一件事？
- [ ] 参数数量是否 <= 3 个？
- [ ] 是否避免了布尔参数？
- [ ] 函数是否有副作用？

### 代码组织

- [ ] 相关代码是否放在一起？
- [ ] 是否遵循项目目录结构？
- [ ] 导入是否按标准库、第三方、本地分组？

---

## 🔧 SOLID 原则检查

### 单一职责原则 (SRP)

- [ ] 每个类是否只有一个变更理由？
- [ ] 每个函数是否只负责一个功能？

### 开放封闭原则 (OCP)

- [ ] 新功能是否通过扩展而非修改实现？
- [ ] 是否使用了策略模式/工厂模式？

### 里氏替换原则 (LSP)

- [ ] 子类是否可以替换父类？
- [ ] 子类是否遵循父类契约？

### 接口隔离原则 (ISP)

- [ ] 接口是否足够小？
- [ ] 客户端是否只依赖需要的接口？

### 依赖倒置原则 (DIP)

- [ ] 高层模块是否依赖抽象？
- [ ] 是否使用依赖注入？

---

## 🏗️ Clean Architecture 检查

### 分层架构

- [ ] 依赖是否指向内层？
- [ ] 业务逻辑是否独立于框架？
- [ ] 数据库访问是否通过 Repository？

### 项目结构

```
src/
├── api/          # 接口层 (Controllers, Routes)
├── core/         # 核心层 (业务逻辑, State)
├── services/     # 服务层 (外部集成)
└── models/       # 数据模型 (ORM)
```

---

## 🧪 TDD 检查

### 测试质量

- [ ] 是否在实现之前编写测试？
- [ ] 测试是否独立且可重复？
- [ ] 测试名称是否描述行为？
- [ ] 是否覆盖边界条件？
- [ ] 是否覆盖错误路径？

### 测试覆盖

- [ ] 单元测试覆盖率 >= 80%？
- [ ] 关键路径是否有集成测试？
- [ ] 新功能是否有对应测试？

---

## 🔒 安全检查

### 输入验证

- [ ] 是否验证所有外部输入？
- [ ] 是否使用参数化查询？
- [ ] 是否避免 SQL 注入？

### 敏感信息

- [ ] 是否避免硬编码密钥？
- [ ] 敏感信息是否使用环境变量？
- [ ] 日志是否避免敏感信息？

### 认证授权

- [ ] API 端点是否有认证保护？
- [ ] 是否使用 HTTPS？
- [ ] 是否实现了适当的权限检查？

---

## ⚡ 性能检查

### 数据库

- [ ] 是否避免 N+1 查询？
- [ ] 是否使用适当的索引？
- [ ] 批量操作是否使用批量插入？

### 异步处理

- [ ] I/O 操作是否使用 async？
- [ ] 是否避免阻塞操作？
- [ ] 是否使用连接池？
- [ ] **async 函数中是否调用了同步阻塞操作？** ← 本次 Bug 的根因
- [ ] 数据库查询是否在 async 函数中直接调用（应该用 `asyncio.to_thread`）？
- [ ] 异常是否被正确传播（不是被静默吞掉）？
- [ ] 返回值是否被正确处理？

---

## 📝 文档检查

### 代码注释

- [ ] 复杂逻辑是否有注释？
- [ ] 注释是否解释"为什么"？
- [ ] 是否删除了注释掉的代码？

### API 文档

- [ ] 新 API 是否有文档？
- [ ] 参数和返回值是否描述清楚？
- [ ] 是否更新了 README？

---

## ✅ 工具检查

### Linting

```bash
# Python
ruff check src tests
```

- [ ] 0 errors
- [ ] 0 warnings (或已记录技术债务)

### 格式化

```bash
ruff format --check src tests
```

- [ ] 所有文件格式化通过

### 类型检查

```bash
pyright src
```

- [ ] 0 errors
- [ ] warnings 已审查

### 测试

```bash
pytest
```

- [ ] 所有测试通过
- [ ] 无跳过的测试（或已记录原因）

---

## 🚀 提交前最终检查

- [ ] 代码是否可运行？
- [ ] 是否有 TODO/FIXME 需要处理？
- [ ] 是否更新了相关文档？
- [ ] 提交信息是否清晰？

---

## 质量门禁规则

### 增量模式 (Incremental)

适用于维护项目，遵循**童子军规则**：

- **新文件**: 必须通过所有 linter 检查（0 错误 0 警告）
- **修改文件**: 警告数必须减少（至少减少 1 个），不允许增加
- **警告数 ≤ 5**: 必须全部修复
