## 一、项目概述

### 1.1 项目定位

Dialog Survey 是一个 AI 驱动的异步多轮对话调研机器人，通过钉钉（DingTalk）平台与受访者进行自然语言对话，收集深度反馈信息。定位为传统问卷调查的补充——比问卷更深入，比人工访谈覆盖更广。

### 1.2 技术栈

| 层 | 技术选型 |
|---|---|
| 运行时 | Node.js + TypeScript (strict mode) |
| Web 框架 | Fastify 5 |
| ORM | Prisma 5.22 → 7.x (评估中) |
| 数据库 | PostgreSQL |
| LLM | OpenAI-compatible API (火山引擎/DashScope) |
| 消息平台 | DingTalk Stream (WebSocket) |
| 前端 | Nunjucks + HTMX + Alpine.js |
| 测试 | Vitest 4.x + Playwright |
| CI/CD | GitHub Actions |
| 代码质量 | Biome (lint + format) + xp-gate |

## 四、经验教训

### 4.1 做得好的地方 ✅

#### 1. 快速迭代，持续交付
- 4个月内发布了约20个版本，平均每周1-2个版本
- 每个版本都有明确的变更内容和版本号
- 用户能持续看到进展，反馈循环短

#### 2. 测试覆盖扎实
- 从0到935个测试用例，覆盖单元、集成、E2E
- 覆盖率阈值80%，CI强制检查
- 测试先行(TDD)在核心模块得到执行

#### 3. 安全意识强
- 早期就引入SSRF防护、Prompt注入防护、HMAC安全比较
- CSP策略、API密钥管理、凭据脱敏
- 安全修复有明确的issue追踪

#### 4. 架构决策审慎
- DI改造(PrismaClient注入)彻底解决了测试隔离问题
- 自定义状态机虽然没用LangGraph，但500行代码清晰可控
- 经过评估后决定不迁移LangGraph，避免了不必要的重构

#### 5. AI辅助开发效率高
- 14个OpenCode会话完成了约200个commit
- 并行子代理(explore/librarian/oracle)加速了调研和实现
- Sprint Flow自动化了从规划到发布的全流程

### 4.2 做得不好的地方 ❌

#### 1. 验收测试反复发现问题
**现象**: 端到端验收测试持续两周，总是能发现新问题，有些问题反复出现。

**根因分析**:
- 缺乏自动化E2E测试，依赖人工验收
- 测试基础设施(Playwright E2E)启动太晚（项目第3个月才开始）
- 状态机修改后没有足够的回归测试覆盖

**教训**: 
> **E2E测试应该从第一天就开始建设，而不是等到验收阶段。** 即使只覆盖核心路径，也能大幅减少反复。

#### 2. 数据库默认值变更未迁移
**现象**: `maxFollowups`从2改为5后，已存在的记录仍然是2，导致问题反复修复。

**根因分析**:
- 使用`prisma db push`而非迁移文件
- 只修改了schema默认值，没有创建数据迁移脚本更新已存在记录
- 同一个问题修了至少3次(ses_0e337615, ses_0ba38c90, ses_0ba38c90续)

**教训**:
> **Schema默认值变更必须配合数据迁移脚本。** `prisma db push`只影响新记录，不更新已存在数据。应该在修改默认值时同步编写一次性迁移脚本。

#### 3. 分支管理导致回归
**现象**: PDF/Markdown导出功能之前已修复，但因分支合并错误问题复现。

**根因分析**:
- 多个功能分支并行开发，合并时冲突处理不当
- 缺乏合并后的自动化验证
- 修复代码和测试没有在同一分支

**教训**:
> **每次合并后必须运行完整测试套件。** 考虑使用squash merge减少合并冲突，或在CI中增加合并后验证步骤。

#### 4. 质量门禁形同虚设
**现象**: xp-gate 9道门禁中，5道因工具未安装或脚本缺失而静默SKIP。

**根因分析**:
- `xp-gate init`不安装CLI工具(jscpd, checkov, hadolint等)
- 覆盖率数据长期显示"N/A"但门禁仍PASS
- 门禁配置与实际工具链不匹配(Biome vs ESLint)

**教训**:
> **质量门禁必须可执行才算数。** 应该在CI中验证所有门禁工具可用，不可用的门禁应该FAIL而不是SKIP。

#### 5. 过度依赖人工验收
**现象**: 大量时间花在人工验收测试上，发现问题后修复，再验收，再发现...

**根因分析**:
- Admin UI没有自动化测试
- 缺少Smoke Test快速验证核心路径
- 没有CI on PR（CI只在tag push时运行）

**教训**:
> **自动化是验收的唯一出路。** 人工验收只能发现"已知类型"的问题，自动化测试才能防止"已知问题"反复。

### 4.3 意外发现 💡

#### 1. 自定义引擎 vs LangGraph
经过详细评估，500行的自定义引擎对于2个节点的简单图来说完全够用。LangGraph的核心收益(checkpointing, streaming)已有替代方案，迁移成本(92个测试重写 + 持久化改造)远大于收益。

> **结论: 不要为了用框架而用框架。** 当你的实现足够清晰且满足需求时，迁移到框架的ROI很低。

#### 2. Prisma 7 + PGlite 方案
调研发现`prisma-pglite-bridge`可以让测试在内存PostgreSQL中运行，零外部依赖，生产代码零改动。这解决了集成测试需要真实数据库的痛点。

> **结论: 测试基础设施的升级往往比功能开发更有价值。**

#### 3. AI辅助开发的效率边界
- AI擅长: 代码生成、测试编写、文档撰写、调研分析
- AI不擅长: 理解业务上下文、判断设计取舍、发现隐含需求
- 人工验收仍然是不可替代的环节

## 六、改进建议

### 6.1 下一个项目应该做的

#### 必须做 (Must Do)

1. **Day 1 就建立E2E测试框架**
   - Playwright/Selenium 覆盖核心用户流程
   - CI on PR 阻断未通过测试的合并
   - Smoke Test 作为每次部署的前置检查

2. **数据库变更必须有迁移脚本**
   - 使用 `prisma migrate` 而非 `prisma db push`
   - 默认值变更必须包含数据更新SQL
   - 迁移脚本必须可回滚

3. **CI on PR，不只是 CI on Tag**
   - 每个PR必须通过完整测试套件
   - Branch protection rule 强制CI通过才能合并
   - 覆盖率报告作为PR评论自动发布

4. **质量门禁必须可执行**
   - 所有门禁工具必须在CI中安装
   - 不可用的门禁应该FAIL而非SKIP
   - 定期检查门禁有效性

#### 应该做 (Should Do)

5. **Squash Merge 策略**
   - 减少合并冲突
   - 保持主分支历史清晰
   - 每个PR一个原子commit

6. **变更日志自动化**
   - Conventional Commits 强制
   - 自动生成 CHANGELOG
   - 版本号自动bump

7. **设计文档先行**
   - 每个功能先写设计文档
   - Delphi评审通过后再编码
   - 设计文档包含测试策略

   - 结构化日志 + 链路追踪

## 七、总结

### 7.1 项目成果

Dialog Survey 从一个Python原型发展为一个生产级的AI对话调研系统，具备：
- ✅ 完整的访谈生命周期管理
- ✅ LLM驱动的智能追问
- ✅ 多维度分析报告生成
- ✅ Admin管理界面
- ✅ 批量导入/导出
- ✅ 跨平台部署支持
- ✅ 935个自动化测试

### 7.2 核心教训

> **1. 测试不是可选项，是交付的一部分。** 没有自动化测试的功能等于没有完成。

> **2. 数据迁移和功能代码一样重要。** 改默认值不改数据，等于没改。

> **3. CI/CD是质量的安全网。** 没有CI on PR，每次合并都是赌博。

> **4. 不要为了用框架而用框架。** 500行清晰的自定义代码 > 5000行框架胶水代码。

> **5. AI辅助开发很强大，但人工验收不可替代。** AI加速了编码，但发现隐含需求仍然需要人。

### 7.3 一句话总结

> **项目成功交付了核心功能，但在测试基础设施和质量门禁上欠了债。下一个项目应该 "测试先行、CI先行、迁移先行"。**
