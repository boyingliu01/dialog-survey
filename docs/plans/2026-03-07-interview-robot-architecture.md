## 二、各层详解

### 2.1 接入层

#### 2.1.1 钉钉消息接收（Webhook）

- 接收被访谈者发送的文本消息、语音消息
- 验证消息签名，确保安全性
- 解析消息内容，提取关键信息

#### 2.1.2 钉钉消息发送（API）

- 向被访谈者发送访谈问题
- 发送访谈邀请、提醒
- 支持文本、卡片等消息类型

#### 2.1.3 语音消息处理

- 接收语音消息文件
- 调用ASR服务转换为文字
- 将文字结果传递给对话引擎

---

### 2.2 应用服务层

#### 2.2.1 钉钉机器人适配器

**功能**：

- 消息解析：解析钉钉消息格式，提取文本、语音、用户信息
- 会话管理：管理每个被访谈者的会话状态
- 菜单交互：处理用户在钉钉中的菜单点击操作
- 回调处理：处理钉钉回调事件

**技术选型**：钉钉开放平台SDK

#### 2.2.2 对话引擎（LangGraph）

采用LangGraph实现多轮对话流程控制，核心节点设计：

| 节点 | 功能 | 说明 |
|------|------|------|
| **规划节点** | 初始化访谈 | 加载访谈模板，生成初始问题 |
| **访谈节点** | 主对话循环 | 接收回答，判定是否追问 |
| **分析节点** | 生成报告 | 访谈完成后生成Markdown报告 |
| **结束节点** | 清理状态 | 保存记录，完成会话 |

**图结构**：

```
┌─────────────┐      是      ┌─────────────┐
│  判定是否   │ ──────────▶ │  生成追问   │
│  需要追问   │              │   问题      │
└──────┬──────┘              └──────┬──────┘
       │                              │
       │否                            │
       ▼                              ▼
┌──────────────┐              ┌──────────────┐
│  判定是否    │ ───是───▶   │  发送问题    │
│  全部完成    │              │  给用户      │
└──────┬───────┘              └──────────────┘
       │否
       ▼
┌──────────────┐
│  进入分析    │
│  节点生成报告│
└──────────────┘
```

#### 2.2.3 状态管理

**对话状态**包含：

| 状态字段 | 说明 |
|----------|------|
| session_id | 会话唯一标识 |
| user_id | 被访谈者ID |
| interview_template | 访谈模板ID |
| current_topic | 当前话题索引 |
| conversation_history | 对话历史（多轮） |
| topic_progress | 话题完成进度 |
| status | 会话状态（进行中/已完成/已中断） |

**存储方案**：PostgreSQL数据库持久化

---

### 2.3 AI服务层

#### 2.3.1 主对话模型

**用途**：生成访谈问题、处理回答、推进对话流程

**模型**：通义千问Max（阿里云）

**Prompt模板结构**：

```
你是专业的访谈主持人，负责进行深度访谈。
访谈主题：{topic}
领域背景：{domain_context}
访谈目标：{interview_goal}

当前访谈进度：
{progress}

请根据用户最新的回答，决定下一步操作：
1. 是否需要追问
2. 生成合适的追问问题或下一个话题
```

#### 2.3.2 追问判定模型

**用途**：判断当前回答是否需要追问，如果需要则生成追问问题

**模型**：通义千问Turbo（阿里云）

**判定逻辑**：

```
根据以下维度判断是否需要追问：
1. 回答深度：回答是否充分、具体
2. 信息完整性：是否包含关键信息点
3. 模糊表达：是否有需要澄清的地方
4. 延伸空间：是否有值得深入的方向
```

#### 2.3.3 报告生成模型

**用途**：访谈完成后，综合所有对话内容生成结构化报告

**模型**：通义千问Max（阿里云）

**输出格式**：Markdown结构化报告

#### 2.3.4 语音识别

**服务**：阿里云Fun-ASR

**功能**：

- 语音消息转文字
- 支持长语音（满足15分钟访谈需求）
- 高准确率

---

### 2.4 数据存储层

#### 2.4.1 对话历史

| 表名 | 说明 |
|------|------|
| interviews | 访谈记录主表 |
| conversations | 对话历史详情 |
| messages | 消息详细内容 |

#### 2.4.2 访谈模板

- 存储格式：JSON
- 包含：主题列表、问题模板、领域知识配置

#### 2.4.3 分析报告

- 存储格式：Markdown文件
- 路径规则：`/reports/{interview_id}/{timestamp}.md`

## 四、关键技术设计

### 4.1 智能追问实现

#### 4.1.1 追问判定流程

```
用户回答 → 提取关键信息 → 追问判定模型
                               │
                    ┌──────────┼──────────┐
                    │          │          │
                    ▼          ▼          ▼
               需要追问    部分需要    无需追问
               (深度追问)  (简单追问)  (继续下一话题)
```

#### 4.1.2 追问类型

| 类型 | 触发条件 | 示例 |
|------|----------|------|
| 澄清追问 | 回答模糊或不具体 | "您刚才提到的'一些问题'能具体说说吗？" |
| 深度追问 | 回答有深度挖掘价值 | "您提到质量有提升空间，能举个例子说明吗？" |
| 验证追问 | 回答前后不一致 | "您前面说A产品好，现在说有问题，能详细解释吗？" |
| 扩展追问 | 回答涉及新话题 | "您提到服务态度好，这在您选择供应商时占多大比重？" |

#### 4.1.3 领域知识注入

通过以下方式实现领域限定：

1. **System Prompt限定**：明确告知LLM只在特定领域内追问
2. **领域知识库**：可选的RAG增强，提供专业背景知识
3. **模板化配置**：预设该领域常见问题类型和追问模式

### 4.2 多轮上下文管理

#### 4.2.1 上下文结构

```json
{
  "session_id": "xxx",
  "interview_template": {
    "topics": [
      {"id": 1, "name": "产品质量", "questions": [...]},
      {"id": 2, "name": "服务质量", "questions": [...]}
    ]
  },
  "current_topic_index": 0,
  "conversation_history": [
    {
      "role": "assistant",
      "content": "欢迎参加访谈，第一个问题：您对我们产品的整体质量满意吗？"
    },
    {
      "role": "user", 
      "content": "总体还行，但有些细节可以改进"
    },
    {
      "role": "assistant", 
      "content": "能具体说说哪些细节吗？"
    },
    {
      "role": "user",
      "content": "比如包装容易破损"
    }
  ],
  "extracted_info": {
    "quality_overall": "还行",
    "improvement_areas": ["包装"]
  }
}
```

#### 4.2.2 上下文持久化

- 每次对话后自动保存到PostgreSQL
- 支持跨会话恢复：下次用户发消息时自动加载历史上下文
- 清理机制：访谈完成后可选择归档或删除

### 4.3 语音处理流程

```
语音消息 → 下载音频 → Fun-ASR转录 → 文字结果 → 对话引擎
```

**技术要点**：

- 音频格式转换：统一转换为16kHz单声道
- 长语音处理：支持分段识别后拼接
- 错误处理：转录失败时提示用户使用文字

## 五、部署方案

### 5.1 基础设施

| 组件 | 规格 | 说明 |
|------|------|------|
| 服务器 | 2核4G起 | 可申请阿里云ECS |
| 数据库 | PostgreSQL | 阿里云RDS或自建 |
| 域名 | 1个 | 用于钉钉回调配置 |

- **语言**：Python 3.10+
- **主要依赖**：
  - langgraph：对话流程控制
  - langchain：LLM编排
  - sqlalchemy：数据库ORM

## 十、总结

本架构设计遵循以下原则：

1. **模块化**：各层职责清晰，便于维护和扩展
2. **可观测**：关键节点可监控，便于问题排查
3. **低成本**：优先使用开源组件，控制API调用成本
4. **安全可靠**：消息验证、数据加密、访问控制
5. **灵活定制**：模板化配置，支持领域知识注入

通过本架构，可以快速实现一个具备智能追问、多轮上下文、语音支持的AI访谈机器人，满足企业内外部深度访谈的需求。
